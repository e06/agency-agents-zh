# 🚀 运行手册：初创企业 MVP 构建

> **模式**: NEXUS-冲刺 | **周期**: 4-6 周 | **智能体**: 18-22 个

---

## 场景

你正在构建一个初创企业 MVP —— 一个需要快速验证产品市场契合度的新产品。速度很重要，质量也同样重要。你需要在 4-6 周内从创意到上线产品并拥有真实用户。

## 智能体阵容

### 核心团队（始终活跃）
| 智能体 | 角色 |
|-------|------|
| Agents Orchestrator（智能体编排器） | 流程控制器 |
| Senior Project Manager（高级项目经理） | 规格转任务 |
| Sprint Prioritizer（冲刺优先级管理器） | 待办事项管理 |
| UX Architect（UX 架构师） | 技术基础 |
| Frontend Developer（前端开发） | UI 实现 |
| Backend Architect（后端架构师） | API 和数据库 |
| DevOps Automator（DevOps 自动化） | CI/CD 和部署 |
| Evidence Collector（证据收集器） | 每个任务的 QA |
| Reality Checker（现实检验器） | 最终质量关卡 |

### 增长团队（第 3 周起激活）
| 智能体 | 角色 |
|-------|------|
| Growth Hacker（增长黑客） | 获客策略 |
| Content Creator（内容创作者） | 发布内容 |
| Social Media Strategist（社交媒体策略师） | 社交营销活动 |

### 支持团队（按需调用）
| 智能体 | 角色 |
|-------|------|
| Brand Guardian（品牌守护者） | 品牌识别 |
| Analytics Reporter（分析报告员） | 指标和仪表盘 |
| Rapid Prototyper（快速原型师） | 快速验证实验 |
| AI Engineer（AI 工程师） | 如果产品包含 AI 功能 |
| Performance Benchmarker（性能基准测试） | 发布前负载测试 |
| Infrastructure Maintainer（基础设施维护） | 生产环境配置 |

## 逐周执行计划

### 第 1 周：发现 + 架构（第 0 阶段 + 第 1 阶段压缩）

```
Day 1-2: 压缩发现
├── Trend Researcher → 快速竞争扫描（1 天，非完整报告）
├── UX Architect → 关键用户流程线框图
└── Senior Project Manager → 将规格转换为任务列表

Day 3-4: 架构
├── UX Architect → CSS 设计系统 + 组件架构
├── Backend Architect → 系统架构 + 数据库模式
├── Brand Guardian → 快速品牌基础（色彩、排版、语调）
└── Sprint Prioritizer → RICE 评分待办事项 + 冲刺计划

Day 5: 基础搭建
├── DevOps Automator → CI/CD 流水线 + 环境配置
├── Frontend Developer → 项目脚手架
├── Backend Architect → 数据库 + API 脚手架
└── Quality Gate: 架构方案获批
```

### 第 2-3 周：核心构建（第 2 阶段 + 第 3 阶段）

```
Sprint 1 (第 2 周):
├── Agents Orchestrator 管理 开发↔QA 循环
├── Frontend Developer → 核心 UI（认证、主视图、导航）
├── Backend Architect → 核心 API（认证、CRUD、业务逻辑）
├── Evidence Collector → 对每个完成任务进行 QA
├── AI Engineer → ML 功能（如适用）
└── 周末冲刺评审

Sprint 2 (第 3 周):
├── 继续剩余功能的 开发↔QA 循环
├── Growth Hacker → 设计病毒传播机制 + 推荐系统
├── Content Creator → 开始创建发布内容
├── Analytics Reporter → 设置追踪和仪表盘
└── 周末冲刺评审
```

### 第 4 周：打磨 + 加固（第 4 阶段）

```
Day 1-2: 质量冲刺
├── Evidence Collector → 完整截图套件
├── Performance Benchmarker → 负载测试
├── Frontend Developer → 修复 QA 问题
├── Backend Architect → 修复 API 问题
└── Brand Guardian → 品牌一致性审计

Day 3-4: 现实检验
├── Reality Checker → 最终集成测试
├── Infrastructure Maintainer → 生产就绪检查
└── DevOps Automator → 生产部署准备

Day 5: 关卡决策
├── Reality Checker 裁决
├── 如需修改: 快速修复周期（2-3 天）
├── 如已就绪: 进入发布阶段
└── Executive Summary Generator → 利益相关者简报
```

### 第 5-6 周：发布 + 增长（第 5 阶段）

```
第 5 周: 发布
├── DevOps Automator → 生产部署
├── Growth Hacker → 激活获客渠道
├── Content Creator → 发布发布内容
├── Social Media Strategist → 跨平台营销活动
├── Analytics Reporter → 实时监控
└── Support Responder → 用户支持上线

第 6 周: 优化
├── Growth Hacker → 分析和优化渠道
├── Feedback Synthesizer → 收集早期用户反馈
├── Experiment Tracker → 启动 A/B 测试
├── Analytics Reporter → 第 1 周分析
└── Sprint Prioritizer → 规划迭代冲刺
```

## 关键决策

| 决策点 | 时间 | 决策者 |
|---------------|------|-------------|
| 概念通过/不通过 | 第 2 天结束 | Studio Producer |
| 架构审批 | 第 4 天结束 | Senior Project Manager |
| MVP 功能范围 | 冲刺规划时 | Sprint Prioritizer |
| 生产就绪 | 第 4 周第 5 天 | Reality Checker |
| 发布时机 | Reality Checker 确认就绪后 | Studio Producer |

## 成功标准

| 指标 | 目标 |
|--------|--------|
| 产品上线时间 | ≤ 6 周 |
| 核心功能完成度 | MVP 范围的 100% |
| 首批用户入驻 | 发布后 48 小时内 |
| 系统正常运行时间 | 首周 > 99% |
| 用户反馈收集 | 前 2 周内 ≥ 50 份回复 |

## 常见陷阱与应对措施

| 陷阱 | 应对措施 |
|---------|-----------|
| 构建过程中范围蔓延 | Sprint Prioritizer 执行 MoSCoW 原则 — "不做"就是不做 |
| 为规模过度工程化 | Rapid Prototyper 思维 — 先验证，后扩展 |
| 为求速度跳过 QA | Evidence Collector 对每个任务运行 — 无例外 |
| 无监控发布 | Infrastructure Maintainer 在第 1 周设置监控 |
| 无反馈机制 | 分析 + 反馈收集内置到冲刺 1 |
