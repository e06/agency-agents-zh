# 🚨 运维手册：事件响应

> **模式**: NEXUS-Micro | **持续时间**: 数分钟至数小时 | **智能体**: 3-8个

---

## 场景

生产环境出现问题。用户受到影响。响应速度很重要，但正确处理同样重要。本运维手册涵盖从检测到事后复盘的全过程。

## 严重程度分类

| 级别 | 定义 | 示例 | 响应时间 |
|-------|-----------|----------|--------------|
| **P0 — 紧急** | 服务完全中断、数据丢失、安全漏洞 | 数据库损坏、DDoS攻击、认证系统故障 | 立即响应（全员出动） |
| **P1 — 高** | 主要功能故障、性能严重下降 | 支付处理中断、错误率超过50%、延迟增加10倍 | < 1小时 |
| **P2 — 中** | 次要功能故障、有可用替代方案 | 搜索功能失效、非关键API错误 | < 4小时 |
| **P3 — 低** | 外观问题、轻微不便 | 样式错误、错别字、小范围UI故障 | 下一迭代 |

## 按严重程度划分的响应团队

### P0 — 紧急响应团队
| 智能体 | 角色 | 行动 |
|-------|------|--------|
| **基础设施维护者** | 事件指挥官 | 评估范围、协调响应 |
| **DevOps自动化工程师** | 部署/回滚 | 如需要则执行回滚 |
| **后端架构师** | 根因调查 | 诊断系统问题 |
| **前端开发工程师** | 前端调查 | 诊断客户端问题 |
| **支持响应专员** | 用户沟通 | 状态页面更新、用户通知 |
| **执行摘要生成器** | 利益相关者沟通 | 实时管理层更新 |

### P1 — 高优先级响应团队
| 智能体 | 角色 |
|-------|------|
| **基础设施维护者** | 事件指挥官 |
| **DevOps自动化工程师** | 部署支持 |
| **相关开发智能体** | 修复实施 |
| **支持响应专员** | 用户沟通 |

### P2 — 中优先级响应
| 智能体 | 角色 |
|-------|------|
| **相关开发智能体** | 修复实施 |
| **证据收集专员** | 验证修复 |

### P3 — 低优先级响应
| 智能体 | 角色 |
|-------|------|
| **迭代优先级管理者** | 添加到待办事项 |

## 事件响应流程

### 步骤 1：检测与分流（0-5分钟）

```
触发条件：监控告警 / 用户报告 / 智能体检测

基础设施维护者：
1. 确认告警
2. 评估范围和影响
   - 影响多少用户？
   - 哪些服务受影响？
   - 数据是否有风险？
3. 分类严重程度（P0/P1/P2/P3）
4. 激活相应的响应团队
5. 创建事件通道/线程

输出：事件分类 + 响应团队已激活
```

### 步骤 2：调查（5-30分钟）

```
并行调查：

基础设施维护者：
├── 检查系统指标（CPU、内存、网络、磁盘）
├── 查看错误日志
├── 检查最近的部署
└── 验证外部依赖

后端架构师（如P0/P1）：
├── 检查数据库健康状态
├── 查看API错误率
├── 检查服务通信
└── 定位故障组件

DevOps自动化工程师：
├── 查看最近的部署历史
├── 检查CI/CD流水线状态
├── 准备回滚（如需要）
└── 验证基础设施状态

输出：已识别根本原因（或已缩小到组件级别）
```

### 步骤 3：缓解（15-60分钟）

```
决策树：

IF 由最近部署引起：
  → DevOps自动化工程师：执行回滚
  → 基础设施维护者：验证恢复
  → 证据收集专员：确认修复

IF 由基础设施问题引起：
  → 基础设施维护者：扩容/重启/故障转移
  → DevOps自动化工程师：支持基础设施变更
  → 验证恢复

IF 由代码缺陷引起：
  → 相关开发智能体：实施热修复
  → 证据收集专员：验证修复
  → DevOps自动化工程师：部署热修复
  → 基础设施维护者：监控恢复

IF 由外部依赖引起：
  → 基础设施维护者：启用降级方案/缓存
  → 支持响应专员：向用户沟通
  → 监控外部恢复

全程：
  → 支持响应专员：每15分钟更新状态页面
  → 执行摘要生成器：向利益相关者简报（仅P0）
```

### 步骤 4：解决验证（修复后）

```
证据收集专员：
1. 验证修复是否解决问题
2. 截图记录正常工作状态
3. 确认未引入新问题

基础设施维护者：
1. 验证所有指标恢复正常
2. 确认无级联故障
3. 修复后监控30分钟

API测试工程师（如涉及API）：
1. 对受影响端点运行回归测试
2. 验证响应时间已正常化
3. 确认错误率恢复基线

输出：事件已解决确认
```

### 步骤 5：事后复盘（48小时内）

```
工作流优化师主导事后复盘：

1. 时间线重建
   - 问题何时引入？
   - 何时被发现？
   - 何时被解决？
   - 用户受影响的总时长

2. 根本原因分析
   - 什么失败了？
   - 为什么失败？
   - 为什么没有更早发现？
   - 5个为什么分析

3. 影响评估
   - 受影响用户
   - 收入影响
   - 声誉影响
   - 数据影响

4. 预防措施
   - 什么监控能更早发现问题？
   - 什么测试能预防此问题？
   - 需要什么流程变更？
   - 需要什么基础设施变更？

5. 行动项
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]

输出：事后复盘报告 → 迭代优先级管理者将预防任务添加到待办事项
```

## 沟通模板

### 状态页面更新（支持响应专员）
```
[时间戳] — [服务名称] 事件

状态：[调查中 / 已识别 / 监控中 / 已解决]
影响：[用户影响描述]
当前行动：[我们正在采取的措施]
下次更新：[预计下次更新时间]
```

### 管理层更新（执行摘要生成器 — 仅P0）
```
事件简报 — [时间戳]

情况：[服务] [中断/降级]，影响 [N名用户/%流量]
原因：[已知/调查中] — [如已知则简述]
行动：[正在采取的措施] — 预计完成时间 [时间预估]
影响：[业务影响 — 收入、用户、声誉]
下次更新：[时间戳]
```

## 升级矩阵

| 条件 | 升级至 | 行动 |
|-----------|------------|--------|
| P0在30分钟内未解决 | 工作室制作人 | 调配更多资源、供应商升级 |
| P1在2小时内未解决 | 项目守护者 | 资源重新分配 |
| 疑似数据泄露 | 法律合规检查员 | 监管通知评估 |
| 用户数据受影响 | 法律合规检查员 + 执行摘要生成器 | GDPR/CCPA通知 |
| 收入影响超过$X | 财务追踪器 + 工作室制作人 | 业务影响评估 |
