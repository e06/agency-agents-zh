# 🛡️ 第四阶段手册 — 质量与强化

> **周期**：3-7 天 | **智能体**：8 个 | **守门人**：现实核查员（唯一权威）

---

## 目标

最后的质量关卡。现实核查员的默认判定是"需要修改"——你必须用充分的证据证明生产就绪。这个阶段的存在是因为初次实现通常需要2-3轮修订，这是健康的。

## 前置条件

- [ ] 第三阶段质量门已通过（所有任务已完成QA）
- [ ] 已收到第三阶段交接包
- [ ] 所有功能已实现并单独验证

## 关键心态

> **现实核查员的默认判定是"需要修改"。**
> 
> 这不是悲观——而是现实。生产就绪需要：
> - 完整的用户旅程端到端可用
> - 跨设备一致性（桌面、平板、移动端）
> - 负载下的性能（不仅仅是理想路径）
> - 安全验证（不仅仅是"我们添加了认证"）
> - 规格合规性（每一项要求，而不是大多数）
>
> 首次通过获得B/B+评级是正常且预期的结果。

## 智能体激活序列

### 步骤 1：证据收集（第1-2天，全部并行）

#### 📸 证据收集员 — 全面视觉证据
```
激活证据收集员为 [项目] 收集全面的系统证据。

所需交付物：
1. 完整截图套件：
   - 桌面端 (1920x1080) — 每个页面/视图
   - 平板端 (768x1024) — 每个页面/视图
   - 移动端 (375x667) — 每个页面/视图
2. 交互证据：
   - 导航流程（点击前后）
   - 表单交互（空白、已填写、已提交、错误状态）
   - 模态框/对话框交互
   - 手风琴/可展开内容
3. 主题证据：
   - 浅色模式 — 所有页面
   - 深色模式 — 所有页面
   - 系统偏好检测
4. 错误状态证据：
   - 404 页面
   - 表单验证错误
   - 网络错误处理
   - 空状态

格式：包含 test-results.json 的截图证据包
时间线：2 天
```

#### 🔌 API测试员 — 完整API回归
```
激活API测试员为 [项目] 进行完整的API回归测试。

所需交付物：
1. 端点回归套件：
   - 所有端点测试 (GET, POST, PUT, DELETE)
   - 认证/授权验证
   - 输入验证测试
   - 错误响应验证
2. 集成测试：
   - 跨服务通信
   - 数据库操作验证
   - 外部API集成
3. 边缘情况测试：
   - 速率限制行为
   - 大负载处理
   - 并发请求处理
   - 格式错误的输入处理

格式：每个端点包含通过/失败状态的API测试报告
时间线：2 天
```

#### ⚡ 性能基准测试员 — 负载测试
```
激活性能基准测试员为 [项目] 进行负载测试。

所需交付物：
1. 10倍预期流量的负载测试：
   - 响应时间分布 (P50, P95, P99)
   - 负载下的吞吐量
   - 负载下的错误率
   - 资源利用率 (CPU, 内存, 网络)
2. 核心Web指标测量：
   - LCP (最大内容绘制) < 2.5s
   - FID (首次输入延迟) < 100ms
   - CLS (累积布局偏移) < 0.1
3. 数据库性能：
   - 查询执行时间
   - 连接池利用率
   - 索引有效性
4. 压力测试结果：
   - 断点识别
   - 优雅降级行为
   - 过载后的恢复时间

格式：性能认证报告
时间线：2 天
```

#### ⚖️ 法律合规检查员 — 最终合规审计
```
激活法律合规检查员为 [项目] 进行最终合规审计。

所需交付物：
1. 隐私合规验证：
   - 隐私政策准确性
   - 同意管理功能
   - 数据主体权利实现
   - Cookie同意实现
2. 安全合规：
   - 数据加密（静态和传输中）
   - 认证安全
   - 输入清理
   - OWASP Top 10 检查
3. 监管合规：
   - GDPR 要求（如适用）
   - CCPA 要求（如适用）
   - 行业特定要求
4. 无障碍合规：
   - WCAG 2.1 AA 验证
   - 屏幕阅读器兼容性
   - 键盘导航

格式：合规认证报告
时间线：2 天
```

### 步骤 2：分析（第3-4天，并行，在步骤1之后）

#### 📊 测试结果分析员 — 质量指标汇总
```
激活测试结果分析员为 [项目] 进行质量指标汇总。

输入：所有步骤1的报告
所需交付物：
1. 汇总质量仪表板：
   - 整体质量评分
   - 分类细分（视觉、功能、性能、安全、合规）
   - 问题严重程度分布
   - 趋势分析（如有多轮测试周期）
2. 问题优先级排序：
   - 关键问题（上线前必须修复）
   - 高优先级问题（上线前应该修复）
   - 中等问题（下个冲刺修复）
   - 低优先级问题（放入待办列表）
3. 风险评估：
   - 生产就绪概率
   - 剩余风险领域
   - 建议的缓解措施

格式：质量指标仪表板
时间线：1 天
```

#### 🔄 工作流优化员 — 流程效率评审
```
激活工作流优化员为 [项目] 进行流程效率评审。

输入：第三阶段执行数据 + 步骤1发现
所需交付物：
1. 流程效率分析：
   - 开发↔QA循环效率（首次通过率、平均重试次数）
   - 瓶颈识别
   - 不同问题类型的解决时间
2. 改进建议：
   - 第六阶段运营的流程变更
   - 自动化机会
   - 质量改进建议

格式：优化建议报告
时间线：1 天
```

#### 🏗️ 基础设施维护员 — 生产就绪检查
```
激活基础设施维护员为 [项目] 进行生产就绪检查。

所需交付物：
1. 生产环境验证：
   - 所有服务健康且响应正常
   - 自动扩缩容已配置并测试
   - 负载均衡器配置已验证
   - SSL/TLS 证书有效
2. 监控验证：
   - 所有关键指标正在收集
   - 告警规则已配置并测试
   - 仪表板访问已验证
   - 日志聚合正常工作
3. 灾难恢复验证：
   - 备份系统正常运行
   - 恢复程序已文档化并测试
   - 故障转移机制已验证
4. 安全验证：
   - 防火墙规则已审查
   - 访问控制已验证
   - 密钥管理已确认
   - 漏洞扫描清洁

格式：基础设施就绪报告
时间线：1 天
```

### 步骤 3：最终裁决（第5-7天，顺序执行）

#### 🔍 现实核查员 — 最终判定
```
激活现实核查员为 [项目] 进行最终集成测试。

强制流程 — 不可跳过：

步骤 1：现实核查命令
- 验证实际构建的内容（ls、grep 查找声称的功能）
- 将声称的功能与规格进行交叉核对
- 运行全面的截图捕获
- 审查步骤1和步骤2的所有证据

步骤 2：QA交叉验证
- 审查证据收集员的发现
- 与API测试员的结果进行交叉引用
- 验证性能基准测试员的数据
- 确认法律合规检查员的发现

步骤 3：端到端系统验证
- 测试完整的用户旅程（而非单个功能）
- 验证所有设备的响应式行为
- 端到端检查交互流程
- 审查实际性能数据

步骤 4：规格现实核查
- 引用原始规格中的确切文本
- 与实际实现证据进行比较
- 记录规格与现实之间的每一个差距
- 不做假设 — 仅凭证据

判定选项：
- 就绪：有充分证据证明生产就绪（首次通过罕见）
- 需要修改：已识别具体问题并提供修复列表（预期结果）
- 未就绪：存在需要回到第一/二阶段的重大架构问题

格式：基于现实的集成报告
默认：需要修改，除非证明否则
```

## 质量门 — 最终关卡

| # | 标准 | 阈值 | 所需证据 |
|---|------|------|----------|
| 1 | 用户旅程完整 | 所有关键路径端到端可用 | 现实核查员截图 |
| 2 | 跨设备一致性 | 桌面 + 平板 + 移动端全部正常 | 响应式截图 |
| 3 | 性能认证 | P95 < 200ms, LCP < 2.5s, 正常运行时间 > 99.9% | 性能基准测试员报告 |
| 4 | 安全验证 | 零关键漏洞 | 安全扫描 + 合规报告 |
| 5 | 合规认证 | 满足所有监管要求 | 法律合规检查员报告 |
| 6 | 规格合规 | 100%规格要求已实现 | 逐点验证 |
| 7 | 基础设施就绪 | 生产环境已验证 | 基础设施维护员报告 |

## 关卡决策

**唯一权威**：现实核查员

### 如果就绪（进入第五阶段）：
```markdown
## 第四阶段 → 第五阶段交接包

### 致发布团队：
- 现实核查员认证报告
- 性能认证
- 合规认证
- 基础设施就绪报告
- 已知限制（如有）

### 致增长黑客：
- 产品已准备好迎接用户
- 用于营销信息的功能列表
- 用于建立可信度的性能数据

### 致DevOps自动化员：
- 生产部署已批准
- 蓝绿部署计划
- 回滚程序已确认
```

### 如果需要修改（返回第三阶段）：
```markdown
## 第四阶段 → 第三阶段返回包

### 修复列表（来自现实核查员）：
1. [关键问题 1]：[描述 + 证据 + 修复说明]
2. [关键问题 2]：[描述 + 证据 + 修复说明]
3. [高优先级问题 1]：[描述 + 证据 + 修复说明]
...

### 流程：
- 问题进入开发↔QA循环（第三阶段机制）
- 每个修复必须通过证据收集员QA
- 当所有修复完成 → 返回第四阶段步骤3
- 现实核查员用更新后的证据重新评估

### 预期：2-3轮修订周期是正常的
```

### 如果未就绪（返回第一/二阶段）：
```markdown
## 第四阶段 → 第一/二阶段返回包

### 已识别的架构问题：
1. [根本性问题]：[为什么无法在第三阶段修复]
2. [结构性问题]：[需要在架构层面改变什么]

### 建议行动：
- [ ] 修订系统架构（第一阶段）
- [ ] 重建基础（第二阶段）
- [ ] 削减范围并重新定义（第一阶段）

### 需要工作室制作人决策
```

---

*第四阶段在现实核查员以充分证据发布"就绪"判定时完成。"需要修改"是预期的首次通过结果——这意味着系统正在工作但需要打磨。*
