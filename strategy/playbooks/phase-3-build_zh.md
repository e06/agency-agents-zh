# 🔨 第三阶段剧本 —— 构建与迭代

> **周期**：2-12 周（根据范围变化）| **智能体**：15-30+ | **守门人**：智能体协调器

---

## 目标

通过持续的「开发↔质量保证」循环实现所有功能。每个任务在下一个任务开始前都经过验证。这是大量工作发生的阶段——也是 NEXUS 协调发挥最大价值的地方。

## 前置条件

- [ ] 第二阶段质量门已通过（基础已验证）
- [ ] 冲刺优先级排序器的待办事项列表可用，包含 RICE 评分
- [ ] CI/CD 流水线已就绪
- [ ] 设计系统和组件库已就绪
- [ ] API 脚手架和认证系统已就绪

## 开发↔质量保证循环 —— 核心机制

智能体协调器通过以下循环管理每个任务：

```
FOR EACH task IN sprint_backlog (按 RICE 评分排序):

  1. ASSIGN task 给合适的开发者智能体（见分配矩阵）
  2. 开发者 IMPLEMENTS 任务
  3. 证据收集者 TESTS 任务
     - 视觉截图（桌面端、平板端、移动端）
     - 针对验收标准的功能验证
     - 品牌一致性检查
  4. IF 结论 == 通过:
       标记任务完成
       进入下一个任务
     ELIF 结论 == 失败 AND 尝试次数 < 3:
       将质量保证反馈发送给开发者
       开发者 FIXES 具体问题
       返回步骤 3
     ELIF 尝试次数 >= 3:
       ESCALATE 给智能体协调器
       协调器决定：重新分配、分解、延期或接受
  5. UPDATE 流水线状态报告
```

## 并行构建轨道

对于 NEXUS-Full 部署，四条轨道同时运行：

### 轨道 A：核心产品开发
```
管理者：智能体协调器（开发↔质量保证循环）
智能体：前端开发者、后端架构师、AI 工程师、
        移动应用构建师、高级开发者
质量保证：证据收集者、API 测试员、测试结果分析员

冲刺节奏：2 周冲刺
每日：任务实现 + 质量保证验证
冲刺结束：冲刺评审 + 回顾
```

### 轨道 B：增长与营销准备
```
管理者：项目领航员
智能体：增长黑客、内容创作者、社媒策略师、
        应用商店优化师

冲刺节奏：与轨道 A 里程碑对齐
活动：
- 增长黑客 → 设计病毒传播循环和推荐机制
- 内容创作者 → 建立发布内容管道
- 社媒策略师 → 规划跨平台营销活动
- 应用商店优化师 → 准备商店页面（如果是移动应用）
```

### 轨道 C：质量与运营
```
管理者：智能体协调器
智能体：证据收集者、API 测试员、性能基准测试员、
        流程优化师、实验追踪员

持续活动：
- 证据收集者 → 为每个任务截取质量保证截图
- API 测试员 → 为每个 API 任务验证端点
- 性能基准测试员 → 定期负载测试
- 流程优化师 → 识别流程改进机会
- 实验追踪员 → 为已验证功能设置 A/B 测试
```

### 轨道 D：品牌与体验打磨
```
管理者：品牌守护者
智能体：UI 设计师、品牌守护者、视觉叙事师、
        趣味注入师

触发式活动：
- UI 设计师 → 当质量保证发现视觉问题时进行组件优化
- 品牌守护者 → 定期品牌一致性审计
- 视觉叙事师 → 功能完成时创建视觉叙事素材
- 趣味注入师 → 微交互和惊喜时刻
```

## 冲刺执行模板

### 冲刺计划（第 1 天）

```
冲刺优先级排序器激活：
1. 审查带有更新 RICE 评分的待办事项
2. 根据团队速度选择冲刺任务
3. 将任务分配给开发者智能体
4. 识别依赖关系和排序
5. 设定冲刺目标和成功标准

输出：包含任务分配的冲刺计划
```

### 每日执行（第 2 天到第 N-1 天）

```
智能体协调器管理：
1. 当前任务状态检查
2. 开发↔质量保证循环执行
3. 阻塞问题识别与解决
4. 进度跟踪与报告

状态报告格式：
- 今日完成任务：[列表]
- 质量保证中任务：[列表]
- 开发中任务：[列表]
- 阻塞任务：[列表及原因]
- 质量保证通过率：[X/Y]
```

### 冲刺评审（第 N 天）

```
项目领航员主持：
1. 演示已完成功能
2. 审查每个任务的质量保证证据
3. 收集利益相关者反馈
4. 根据所学更新待办事项

参与者：所有活跃智能体 + 利益相关者
输出：冲刺评审总结
```

### 冲刺回顾

```
流程优化师主持：
1. 哪些做得好？
2. 哪些可以改进？
3. 下个冲刺我们将改变什么？
4. 流程效率指标

输出：回顾行动项
```

## 协调器决策逻辑

### 任务失败处理

```
WHEN 任务质量保证失败：
  IF 尝试次数 == 1:
    → 将具体质量保证反馈发送给开发者
    → 开发者仅修复已识别的问题
    → 重新提交质量保证
    
  IF 尝试次数 == 2:
    → 发送累积的质量保证反馈
    → 考虑：开发者智能体是否合适？
    → 开发者带着额外上下文修复
    → 重新提交质量保证
    
  IF 尝试次数 == 3:
    → 升级
    → 选项：
      a) 重新分配给不同的开发者智能体
      b) 将任务分解为更小的子任务
      c) 修改方案/架构
      d) 接受已知限制（记录在案）
      e) 延期到未来冲刺
    → 记录决策和理由
```

### 并行任务管理

```
WHEN 多个任务无依赖关系：
  → 同时分配给不同的开发者智能体
  → 各自运行独立的开发↔质量保证循环
  → 协调器并发跟踪所有循环
  → 按依赖顺序合并已完成任务

WHEN 任务有依赖关系：
  → 等待依赖项通过质量保证
  → 然后分配依赖任务
  → 在交接中包含依赖上下文
```

| 7 | 所有 PR 代码审查已完成 | Git 历史证据 | ☐ |

## 质量门决策

**守门人**：智能体协调器

- **通过**：功能完整的应用程序 → 激活第四阶段
- **继续**：需要更多冲刺 → 继续第三阶段
- **升级**：系统性问题 → 工作室制作人介入
